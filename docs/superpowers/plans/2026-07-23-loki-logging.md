# Loki Logging for `dev` Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add centralized log aggregation (Loki + Promtail) to the `dev` Kustomize environment, wired into the existing Grafana instance, following the design in `docs/superpowers/specs/2026-07-23-loki-logging-design.md`.

**Architecture:** A new `logging/dev/` Kustomize directory uses the `helmCharts` generator (same mechanism as `monitoring/dev/`) to install the `grafana/loki-stack` chart (Loki single-binary + Promtail DaemonSet) into the existing `dev-monitoring` namespace. The chart's own `templates/datasources.yaml` auto-generates a Grafana datasource ConfigMap that the already-configured `kube-prometheus-stack` Grafana sidecar picks up — no hand-written manifests beyond the two Kustomize files. `logging/dev` is added to `clusters/dev/kustomization.yaml`'s resource list, after `monitoring/dev` so the `dev-monitoring` namespace exists first.

**Tech Stack:** Kustomize v5.6.0 (`--enable-helm`), Helm chart `grafana/loki-stack` v2.10.3 (repo `https://grafana.github.io/helm-charts`), kubectl (server-side apply).

## Global Constraints

- Chart version pinned: `grafana/loki-stack` **2.10.3** (matches the pinning convention already used for `kube-prometheus-stack` in `monitoring/dev/kustomization.yaml`).
- Namespace: `dev-monitoring` (existing — created by `monitoring/dev/namespace.yaml`; do not create a second `Namespace` resource for it).
- Persistence: disabled (`loki.persistence.enabled: false`) — no PVC/StorageClass dependency, consistent with this being a sandbox/test-scenario environment.
- Datasource default: `loki.isDefault: false` — must not contest Prometheus's existing default-datasource status in Grafana.
- This repo's convention (confirmed via `git ls-files monitoring/dev/charts`, 301 tracked files) is to **commit the local Helm chart cache** that `kustomize build --enable-helm` writes alongside the kustomization (e.g. `monitoring/dev/charts/kube-prometheus-stack-87.16.1/...`). `logging/dev/charts/...` must be committed the same way, not gitignored.
- Only `dev` is in scope. No `prod` equivalent (matches the rest of this repo).

### Task 1: Scaffold `logging/dev` and verify it renders standalone

**Files:**
- Create: `logging/dev/kustomization.yaml`
- Create: `logging/dev/values.yaml`
- Create (generated by kustomize, then committed): `logging/dev/charts/loki-stack-2.10.3/**`

**Interfaces:**
- Produces: a Kustomize directory buildable on its own via `kustomize build --enable-helm logging/dev`, rendering a `Service/loki` (port 3100), a Loki workload (Deployment or StatefulSet named `loki`), a `DaemonSet/loki-promtail`, and a `ConfigMap` labeled `grafana_datasource: "1"` whose data contains a `Loki` datasource with `url: http://loki:3100`.

- [ ] **Step 1: Write `logging/dev/kustomization.yaml`**

```yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

helmCharts:
    - name: loki-stack
      repo: https://grafana.github.io/helm-charts
      version: 2.10.3
      releaseName: loki
      namespace: dev-monitoring
      valuesFile: values.yaml
      includeCRDs: true
```

### Task 3: Apply to the live `dev` cluster and verify Loki/Promtail health

**Files:** none (cluster-state task only)

**Interfaces:**
- Consumes: `clusters/dev` (Task 2), buildable and collision-free.
- Produces: a live cluster with `loki` and `loki-promtail` pods `Running` in `dev-monitoring`, and Grafana's datasource ConfigMap present.

- [ ] **Step 1: Confirm the kubectl context before touching anything**

```bash
kubectl config current-context
```

This must be the intended local/dev cluster (e.g. a `minikube` profile), never a shared or prod context. If unsure, stop and ask before proceeding — do not guess on a mutating step.

- [ ] **Step 2: Apply using the existing `clusters/dev` procedure**

Follow the `.claude/skills/apply-cluster-dev/SKILL.md` procedure (build → server-side apply → restart the Prometheus Operator once if it logged `not installed` on a cold cluster → re-apply). The rendered file from Task 2, Step 2 (`/tmp/clusters-dev-render.yaml`) already includes the new Loki/Promtail resources, so no separate apply step is needed for them — they ride along with the rest of the tree:


- [ ] **Step 3: Verify Loki and Promtail pods**

```bash
kubectl get pods -n dev-monitoring -l app=loki
kubectl get daemonset loki-promtail -n dev-monitoring
kubectl rollout status daemonset/loki-promtail -n dev-monitoring --timeout=90s
```

Expected: at least one `loki-...` pod `Running`/`1/1 Ready` (Loki's own label is `app: loki`, from the subchart's `loki.name` template), the `loki-promtail` DaemonSet exists (Promtail's subchart labels its pods `app: promtail`, not `app: loki`, so it's checked by object name here instead), and `daemonset "loki-promtail" successfully rolled out`.

Expected: a ConfigMap named `loki` (or `loki-loki-stack`, depending on the chart's `loki-stack.fullname` resolution — record whichever name actually appears) is listed.

- [ ] **Step 5: Confirm Grafana loaded it**


Expected: a log line showing the sidecar wrote/updated a datasource file referencing Loki (the sidecar container logs the filename it wrote on each sync).

### Task 4: End-to-end log validation

**Files:** none (verification-only task)

**Interfaces:**
- Consumes: a live cluster from Task 3 with Loki reachable at `loki.dev-monitoring.svc.cluster.local:3100`, and the `frontend`/`backend`/`cache` services already running in namespace `dev` (pre-existing, unrelated to this plan).
- Produces: confirmation that stdout logs from the three app pods are actually queryable in Loki.

- [ ] **Step 1: Port-forward Loki and generate traffic through the app chain**

```bash
kubectl port-forward svc/loki -n dev-monitoring 3100:3100 &
LOKI_PF_PID=$!
sleep 2

kubectl port-forward svc/frontend-service -n dev 8080:80 &
FRONTEND_PF_PID=$!
sleep 2

curl -s -o /dev/null -w "frontend responded: %{http_code}\n" http://localhost:8080/
```

Expected: `frontend responded: 200` (or whatever success code the test app returns — a non-`000`/non-5xx response confirming the request actually went through frontend → backend → cache).

- [ ] **Step 2: Query Loki for logs from the `dev` namespace**

```bash
START=$(( $(date +%s) - 300 ))000000000
END=$(date +%s)000000000

curl -s -G "http://localhost:3100/loki/api/v1/query_range" \
  --data-urlencode 'query={namespace="dev"}' \
  --data-urlencode "start=${START}" \
  --data-urlencode "end=${END}" \
  --data-urlencode "limit=50" \
  --data-urlencode "direction=backward" \
  | tee /tmp/loki-query-result.json \
  | python3 -c "import json,sys; d=json.load(sys.stdin); print(len(d.get('data',{}).get('result',[])), 'streams')"
```

Expected: `N streams` where `N > 0` — at least one log stream matched. If it prints `0 streams`, check `/tmp/loki-query-result.json` for the raw response and confirm Step 1 actually generated traffic recently enough to fall inside the 5-minute window.

- [ ] **Step 3: Confirm at least one line is attributable to each app service**

```bash
python3 -c "
import json
d = json.load(open('/tmp/loki-query-result.json'))
pods = {s['stream'].get('pod', s['stream'].get('container', '?')) for s in d.get('data', {}).get('result', [])}
print(sorted(pods))
"
```

Expected: a list of pod/container names that includes something recognizable from each of `frontend`, `backend`, and `cache` (exact label naming depends on the chart's Promtail scrape config — record what actually shows up).

- [ ] **Step 4: Clean up the port-forwards**

```bash
kill $LOKI_PF_PID $FRONTEND_PF_PID 2>/dev/null
```

No commit for this task — it's a live-cluster verification task with no repo changes.
